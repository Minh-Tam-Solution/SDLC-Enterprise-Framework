**Purpose**: Flowchart mapping requirements → principles → enforcement actions

## Overview

This decision matrix maps every governance scenario to a concrete enforcement action. When a gate evaluation occurs, the Orchestrator follows this matrix to determine the correct principle and action.

---

## Decision Flow

```
Requirement arrives
       │
       ▼
┌─────────────────────────┐
│ 1. Is this AI-generated? │
└──────┬──────────────────┘
       │
   YES │                    NO → Standard code review (Tier 1/2/3)
       ▼
┌─────────────────────────┐
│ 2. Risk Classification  │
└──────┬──────────────────┘
       │
       ├── LOW (info-level, docs, config)
       │   → Principle 1: AI-Human Collaboration
       │   → Action: AUTO_APPROVE (AI executes, human spot-checks)
       │
       ├── MEDIUM (business logic, UI, tests)
       │   → Principle 3: Planning Mode
       │   → Action: REQUIRE_REVIEW (plan before code, evidence required)
       │
       ├── HIGH (auth, payments, API contracts, data schema)
       │   → Principle 4: Verification-First
       │   → Action: REQUIRE_APPROVAL (TDD mandatory, 2+ reviewers)
       │
       └── CRITICAL (security, compliance, infrastructure)
           → Principle 2: Agent Accountability
           → Action: BLOCK_UNTIL_CTO (human owner signs off)
```

---

## Risk Classification Rules

| Risk Level | Triggers | Principle | Enforcement |
|------------|----------|-----------|-------------|
| **LOW** | Documentation, comments, config files, README | P1: AI-Human Collaboration | Auto-approve, log evidence |
| **MEDIUM** | Business logic (<50 LOC), UI components, unit tests | P3: Planning Mode | Require planning artifact + review |
| **HIGH** | Auth/authz, payment logic, API contracts, DB schema, >50 LOC | P4: Verification-First | TDD required, 2+ approvers, integration tests |
| **CRITICAL** | Security policies, compliance rules, infrastructure, secrets | P2: Agent Accountability | Block until human owner approval (CTO/Security Lead) |

## Vibecoding Index Routing

The Vibecoding Index (0-100) measures AI contribution quality. Routing depends on the score:

| Index Range | Zone | Action | Approval Required |
|-------------|------|--------|-------------------|
| 0-30 | GREEN | Auto-approve | None (logged) |
| 31-60 | YELLOW | Tech Lead review | Tech Lead |
| 61-80 | ORANGE | CEO should review | Senior Engineer + CEO |
| 81-100 | RED | CEO must review | CTO + CEO |

### Vibecoding Index Signals (5 weighted)

| Signal | Weight | Description |
|--------|--------|-------------|
| AI LOC ratio | 25% | Percentage of lines generated by AI |
| Test coverage delta | 25% | Coverage change after AI contribution |
| Review depth | 20% | Human review thoroughness (comments/LOC) |
| Planning artifact | 15% | Whether planning mode was used |
| Context freshness | 15% | AGENTS.md recency (<24h = good) |

## Principle → Enforcement Mapping

### Principle 1: AI-Human Collaboration

**When**: Low-risk AI-generated code
**Enforcement**:
- Gate action: `AUTO_APPROVED`
- Evidence: Commit log with AI attribution
- Audit: Logged, no human review required

### Principle 2: Agent Accountability

**When**: Critical-risk changes OR Vibecoding Index RED zone
**Enforcement**:
- Gate action: `BLOCKED_NEED_CTO_CEO`
- Evidence: Human sign-off required (named individual)
- Audit: Full audit trail, accountability record created

### Principle 3: Planning Mode

**When**: Medium-risk OR >50 LOC changes OR API/schema changes
**Enforcement**:
- Gate action: `NEED_TECH_LEAD_APPROVAL`
- Evidence: Planning artifact (design doc, ADR, or plan file)
- Audit: Planning artifact hash stored in Evidence Vault

### Principle 4: Verification-First

**When**: High-risk changes (auth, payments, data)
**Enforcement**:
- Gate action: `NEED_CEO_APPROVAL` (or `NEED_SENIOR_APPROVAL`)
- Evidence: TDD proof (RED-GREEN-REFACTOR), integration test results
- Audit: Test results + coverage delta stored

### Principle 5: Context Management

**When**: AGENTS.md or context files modified
**Enforcement**:
- Gate action: `WARNED` (context drift check)
- Evidence: Context diff, freshness check
- Audit: Context update logged

### Principle 6: Tool Evaluation

**When**: New AI tool introduced or tool configuration changed
**Enforcement**:
- Gate action: `NEED_TECH_LEAD_APPROVAL`
- Evidence: Tool evaluation scorecard (per 06-Tool-Evaluation-Criteria.md)
- Audit: Evaluation record stored

### Principle 7: Anti-Patterns

**When**: Known anti-pattern detected (blind acceptance, context stuffing, etc.)
**Enforcement**:
- Gate action: `BLOCKED`
- Evidence: Anti-pattern detection report
- Audit: Violation logged, developer notified

## Governance Mode Impact

| Mode | LOW Risk | MEDIUM Risk | HIGH Risk | CRITICAL Risk |
|------|----------|-------------|-----------|---------------|
| **OFF** | Pass | Pass | Pass | Pass |
| **SOFT** | Auto-approve | Warn | Warn | Block |
| **WARNING** | Auto-approve | Warn | Block | Block |
| **FULL** | Auto-approve | Require approval | Require CTO | Block + CTO |

## Implementation in Orchestrator

The `compute_gate_actions()` function (Pre-Phase contract) implements this matrix:

```python
def compute_gate_actions(gate, user, mode):
    risk = classify_risk(gate.exit_criteria)
    vi_score = compute_vibecoding_index(gate)

    if mode == GovernanceMode.OFF:
        return all_actions_allowed()

    if risk == "CRITICAL":
        return block_until_approval("governance:approve")

    if vi_score > 80:  # RED zone
        return block_until_approval("governance:approve")

    if risk == "HIGH" or vi_score > 60:
        return require_verification(reviewers=2, tdd=True)

    if risk == "MEDIUM" or vi_score > 30:
        return require_planning_artifact()

    return auto_approve_with_logging()
```
